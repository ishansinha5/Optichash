# Database Initialization: Spatial Routing & Cache Schema

This directory contains the bootstrap SQL scripts used to initialize the PostgreSQL container when the Docker Compose stack spins up. Rather than manually running database migrations every time the local environment is rebuilt, the `init.sql` file is automatically mapped to the container's `/docker-entrypoint-initdb.d/` directory to construct the exact schema required by the Java API Gateway and the C++ Perceptual Hash layer.

## The Architecture of `init.sql`

When building the persistence layer for OpticHash, I wanted to ensure the database wasn't just a passive storage bucket. It needed to actively support both the spatial store-locator features and the high-speed caching requirements of the compute-optimized vision pipeline. 

Here is the technical reasoning behind the schema design:

* **PostGIS Spatial Extension:** The script begins by explicitly initializing `CREATE EXTENSION IF NOT EXISTS postgis;`. While not fully utilized by the baseline image hashing, installing this extension at boot is critical for Phase 2. It allows PostgreSQL to handle complex geometric coordinate math natively, which will be used to calculate the physical distance between a user's IP/location and the nearest local comic book shops carrying a specific variant.
* **The `locg_variants` Core Table:** Instead of over-normalizing the data into dozens of fragmented sub-tables—which would require expensive SQL `JOIN` operations that slow down response times—I flattened the core variant data into a single, high-read-speed table.
* **The Hardware Bypass Key (`phash`):** The `phash VARCHAR(64) UNIQUE NOT NULL` column is arguably the most important architectural piece of the entire project. This column stores the string representation of the 64-bit integer generated by the C++ perceptual hashing bouncer. Enforcing a `UNIQUE` constraint here guarantees an exact 1:1 mapping for our cache; if the C++ service queries this column and finds a match, the PyTorch neural network is completely bypassed.
* **League of Comic Geeks (LoCG) Parity:** The metadata columns (`publisher`, `series`, `issue_number`, `upc`, `distributor_sku`) are specifically engineered to mirror the exact JSON schema utilized by the League of Comic Geeks. This structural parity ensures that when the REST API is modularized for external webhooks, the data payloads will serialize and deserialize perfectly without requiring a heavy middleware translation layer.
* **Action Hooks:** The boolean fields (`pull_it`, `have_it`, `read_it`, `want_it`) provide the underlying state management for user collections. Setting `DEFAULT false` ensures that new records can be inserted rapidly via the Python inference worker without needing to pass exhaustive insert statements.
* **Optimized AI Telemetry (`compute_cycles_saved`):** To actually prove that the two-stage evaluation matrix is working, the database actively tracks its own efficiency. Every time the C++ layer successfully serves a cached image via the `phash` column, this integer increments. This provides hard, verifiable telemetry on exactly how much deep learning compute—and by extension, electrical overhead—the system is preventing.
